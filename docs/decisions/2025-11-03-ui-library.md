# Decision Record: Playground UI コンポーネント方針

## 背景
- `/playground` のトップページを皮切りに、Mastra 機能ごとにプレイグラウンドページを段階追加する計画がある。
- 製品特性上、モダンなカードレイアウト・セクション分割・バッジ表示などのUIパターンを繰り返し利用する見込み。
- UI コンポーネントは server-only 原則を破らずに再利用でき、カスタマイズのしやすさやデザイン一貫性を両立させる必要がある。

## 選択肢
1. **Tailwind の生クラスを継続利用**
   - 追加依存を増やさず軽量。
   - ただしバリエーションや状態管理を各ページで都度実装する必要があり、保守性が低い。
2. **MUI を導入**
   - 豊富なコンポーネントとアクセシビリティサポート。
   - 既定テーマが濃く、Tailwind ベースの既存スタイルとの整合が難しい。軽量なプレイグラウンドでもクライアントサイドのJSが一定量増える。
3. **shadcn/ui（Tailwind コンポーネント）を導入**
   - Tailwind クラス互換で学習コストが低く、必要なコンポーネントのみコピーできる。
   - `class-variance-authority` や `tailwind-merge` を併用することでバリアント管理が容易。

## 決定
- **shadcn/ui ベースのコンポーネントを導入**し、Tailwind との親和性を保ったまま UI パターンを共通化する。
- カード、ボタン、バッジ等の軽量コンポーネントを `components/ui/` 配下に追加し、Playground ページで即時活用した。
- MUI はフォームやデータテーブルなど複雑UIが必要になった際にページ単位で導入する（併用可）方針を維持する。

## 含意
- 追加した共通コンポーネントは `lib/utils.ts` の `cn` ユーティリティを経由してクラスマージを行う。
- 新規コンポーネントを追加する場合は `shadcn/ui` のガイドラインに沿って `components/ui/` に配置し、zod 型・Story/テスト追加を検討する。
- MUI を採用するページが現れた場合は、docs/decisions 配下に別 Decision Record を作成し、採用理由と scoped import 戦略を記録する。
